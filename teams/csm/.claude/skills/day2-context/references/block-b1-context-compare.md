## EXPLAIN

### 1. 한 문장 요약
맥락은 답의 품질을 바꾸는 게 아니라 질문의 의미 자체를 바꾼다 — "리캡 메일 써줘"는 맥락에 따라 완전히 다른 작업이 된다.

---

### 2. 왜 필요한가 (Why)

**Day 1에서 배운 것의 한계**

Day 1에서 CLAUDE.md를 만들었다. 팀 톤, 금지어, 워크플로우 — 변하지 않는 기본값을 한 번 써뒀다. 그것만으로도 리캡 메일 초안의 품질이 달라졌다.

하지만 Day 1이 끝나고 실제 업무에서 이런 상황이 온다:

```
어제 고객 A와 미팅을 했다.
논의: 대시보드 세팅 이슈 → 해결책 2가지 제안 → 다음 주 확인 합의

"고객 A 리캡 메일 써줘"
→ AI가 무엇을 쓰나?
```

CLAUDE.md에 고객 A의 일반 특성은 있다. 하지만 "어제 어떤 논의가 있었는지"는 없다. AI는 빈 자리를 일반적인 내용으로 채운다. 어제 회의 구체 내용이 반영된 메일이 나올 수 없다.

**변하지 않는 맥락 vs 변하는 맥락**

```
변하지 않는 맥락 (Day 1에서 CLAUDE.md로 해결)
  → 팀 톤 / 금지어 / 워크플로우 기본 구조
  → 한 번 써두면 계속 효과

변하는 맥락 (Day 2에서 해결해야 하는 것)
  → 어제 미팅 노트 / 최근 Slack 대화 / 고객 현재 계약 상태
  → 매번 바뀌므로 실시간으로 가져와야 함
```

Day 2의 핵심 질문: "변하는 맥락을 AI에게 어떻게 전달할 것인가?"

답: MCP (Model Context Protocol) — 외부 시스템의 데이터를 AI가 직접 읽도록 연결한다.

---

### 3. 본질 (What)

**맥락이 바꾸는 건 정답이 아니라 질문의 의미다**

```
[맥락 없을 때 — AI에게 주어진 것: 프롬프트 한 줄]
  "리캡 메일 써줘"

  AI가 모르는 것:
    - 고객 이름
    - 어제 무슨 논의가 있었나
    - 어떤 합의가 있었나
    - 다음 액션은 무엇인가

  AI의 선택: 빈 자리를 일반 패턴으로 채운다
  결과: Hi [Customer], Thanks for the great meeting...
        → 어느 고객에게나 쓸 수 있는 일반 템플릿

[고객 A 맥락 있을 때]
  AI가 아는 것:
    - 고객 이름: 이에나
    - 어제 논의: 대시보드 세팅 이슈 → 해결책 2가지 제안
    - 합의: 다음 주까지 데이터팀 확인 후 공유
    - 다음 액션: 수요일 오후 재확인 미팅

  AI의 선택: 알고 있는 것을 토대로 구체적으로 쓴다
  결과: 이에나님, 어제 대시보드 세팅 함께 검토해주셔서 감사합니다.
        제안드린 두 가지 방법 중 data layer 방식으로 진행하기로 했고,
        다음 주 수요일 오전까지 데이터팀 검토 결과 공유드리겠습니다.
        → 이름 호명 / 구체 합의사항 / 다음 일정
```

"리캡 메일 써줘"라는 동일한 입력이, 맥락에 따라 완전히 다른 작업이 된다.

**두 종류의 맥락을 연결하는 구조**

```
CLAUDE.md (어제 만든 것):
  고정 맥락 — 팀 톤, 금지어, 워크플로우
  → 항상 켜진 상태

MCP (오늘 배울 것):
  변하는 맥락 — 어제 Notion 미팅 노트, 최근 Slack, 현재 계약 상태
  → 필요할 때 실시간으로 가져옴

두 가지가 합쳐졌을 때:
  AI는 팀 기본값 + 실시간 데이터 위에서 작업
  → 메일 초안이 "내 것처럼" 느껴지는 수준
```

### 4. 어떻게 작동하나 (How)

**맥락이 없을 때 AI가 하는 일**

AI는 빈 자리를 채워야 한다. 모르면 추정한다. 추정할 때 쓰는 것은 학습 데이터의 평균 패턴이다. 그래서 맥락 없는 결과가 "무난하지만 내 것이 아닌" 느낌이다. 틀리지 않았지만 나만의 것도 아니다.

**맥락이 있을 때 AI가 하는 일**

AI는 알고 있는 것을 먼저 쓴다. 고객 이름, 어제 논의, 합의사항이 있으면 그것을 중심으로 구체적으로 채운다. 평균 패턴이 아니라 이 특정 상황에 맞는 내용이 나온다.

**Day 2에서 배울 흐름**

```
오늘 B2: MCP가 무엇인지 개념 이해
오늘 B3: 실제로 MCP 1개 연결해보기 (Gmail·Notion·Slack 중 1개)
오늘 B4: Skill 첫 제작 — 반복 작업을 레시피로 만들기
오늘 B5: Git + 팀 repo에 올리기
```

### 5. 흔한 오해 (Common Misconceptions)

**오해 1: "CLAUDE.md가 있으면 맥락 문제가 다 해결된다"**

CLAUDE.md는 "변하지 않는 맥락"을 해결한다. 어제 미팅 내용, 오늘 Slack 대화, 현재 계약 단계는 변하는 맥락이다. 이건 CLAUDE.md로 해결할 수 없다. MCP로 실시간 연결해야 한다.

**오해 2: "맥락을 매번 붙여넣으면 되지, 왜 MCP가 필요한가"**

매번 붙여넣으면 몇 가지 문제가 생긴다. 첫째, 어디서 무엇을 가져올지 찾는 시간이 든다. 둘째, 찾는 도중 중요한 내용을 빠뜨릴 수 있다. 셋째, 고객이 늘수록 이 과정이 비례해서 늘어난다. MCP는 이 과정을 AI가 직접 하게 만든다.

**오해 3: "맥락이 많을수록 무조건 좋다"**

관련 없는 맥락이 많으면 오히려 AI가 혼란스러워한다. 필요한 맥락을 정확하게 가져오는 것이 목표다. Notion 전체를 연결하는 것보다, 이 고객의 최근 미팅 노트만 가져오는 것이 더 좋은 결과를 만든다.

### 6. CSM 일상 적용

**오늘 B3에서 할 것**

Notion, Gmail, Slack 중 가장 자주 쓰는 것 1개를 MCP로 연결한다. 연결 후 "고객 A 최근 미팅 노트 요약해줘"라고 요청해보면, 매번 Notion에서 찾아 붙여넣던 과정이 사라진다.

**기대할 수 있는 변화**

```
Before (MCP 없을 때):
  Notion 열기 → 고객 A 폴더 찾기 → 미팅 노트 열기 → 핵심 복사 → 붙여넣기
  시간: 5~10분

After (MCP 연결 후):
  "고객 A 최근 미팅 노트 요약해줘"
  시간: 30초
```
