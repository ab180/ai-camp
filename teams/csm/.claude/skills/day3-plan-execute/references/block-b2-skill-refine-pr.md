# Block B2: SKILL.md 다듬기 (Claude Code와 대화하며) + 두 번째 PR

> **Phase A 시작 시 반드시 아래 형태로 출력한다:**
> ```
> 📖 공식 문서: https://code.claude.com/docs/ko/skills
> 📖 GitHub PR: https://docs.github.com/ko/pull-requests
> ```

## EXPLAIN

### 4단계 흐름

**1단계: SKILL.md + 단서 나란히 펼치기 (10분)**

```
열어야 할 것 두 가지:
  - 어제 만든 SKILL.md
  - B1에서 받은 스펙 문서 (또는 메모)

나란히 보면서: "스펙에 있는데 SKILL.md에 없는 것"을 확인
```

**2단계: Claude Code와 대화하며 다듬기 (30분) ← 핵심**

> 혼자 Edit으로 박지 말 것.
> "이 항목 추가하고 싶은데 표현은 어떻게 하면 좋을까?" 식으로 같이 풀어 적는다.
> 사람이 결정, AI가 표현·구조를 잡는다.

대화 예시:
```
YOU: SKILL.md에 "분기 체크업 미팅 시 다음 분기 약속을 절대 못 박는다"는
     판단 기준을 추가하고 싶어. 어떤 섹션에 어떤 표현이 좋을까?

CC:  "## 절대 금지"에 두는 게 일반적이에요.
     "다음 분기 일정·약속을 임의로 명시하지 않는다. 영업 협의 후 결정."

YOU: 좋아. "고객이 먼저 묻는 경우"는 예외인데 그건 어떻게 적지?

CC:  같은 섹션 하단에 "## 예외" 추가해서 분리하는 게 깔끔해요.
```

30분 동안 3-4개 항목만. 한 항목당 평균 7-10분.

**3단계: SKILL.md 검토 + 한 번 호출 (10분)**

```
SKILL.md 다시 읽으며:
  □ 이해가 안 되는 문장 없나?
  □ 너무 길거나 모호한 섹션 없나?
  □ 내가 "이건 아닌데" 싶은 표현 없나?

한 번 호출:
  /skill [내 스킬 이름]
  → 결과가 어제보다 나아졌나?
```

**4단계: 두 번째 PR (10분)**

```
git diff SKILL.md               ← 어제와 오늘 차이 확인
git commit -m "refactor: 빠진 절차·판단 추가"
git push
→ GitHub 웹에서 PR 생성
  제목: "refactor: [스킬 이름] 암묵지 추가"
  본문: 이번에 추가한 항목 3-4개 짧게
  리뷰어: 짝꿍 지정
```

### 두 PR이 만드는 흔적

```
PR #1 (어제) — SKILL.md 신규 (스킬 1차)
PR #2 (오늘) — SKILL.md 보강 (암묵지 추가)

git log에 2개 commit
= 어제 vs 오늘 차이 = "암묵지 → 명시지" 변화량
```

## EXECUTE

위 4단계를 순서대로 진행하세요.

B1에서 받은 스펙 문서를 열고, 어제 SKILL.md와 나란히 보며 시작합니다.

**가장 중요한 규칙: 혼자 파일을 직접 편집하지 마세요.**
Claude Code에게 말로 설명하고, Claude가 표현을 잡아주면 "맞아/아니야"로 결정하세요.
