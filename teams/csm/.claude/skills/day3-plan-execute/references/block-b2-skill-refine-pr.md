## EXPLAIN

### 1. 핵심 한 줄
스킬은 한 번에 완성되지 않는다. Claude Code와 대화하면서 다듬고, 그 결과를 PR로 팀과 공유하는 것이 오늘의 두 번째 PR이다.

---

### 2. 슬라이드 흐름 따라 개념 풀이

**스킬 커스터마이징 — 운영진 베이스를 내 것으로**

오늘 스킬 커스터마이징의 원칙: **0에서 시작하지 않는다.** 운영진이 만든 베이스 스킬 3종 중 하나를 골라서 내 맥락으로 수정한다.

베이스 스킬 3종:
- **리캡 메일 스킬**: 미팅 노트 입력 → 리캡 메일 초안
- **문의 답변 스킬**: 문의 본문 입력 → 답변 초안
- **체크업 미팅 자료 스킬**: 고객 정보 입력 → 아젠다 + 지표 요약

스킬 구조:
```
skills/recap-mail/
├── SKILL.md       ← 언제 쓰는 스킬인지, 입력·출력·예시
├── templates/     ← 출력 포맷
└── rules/         ← 톤·금지어·필수 섹션
```

**Claude Code와 대화하면서 다듬는 법**

스킬을 다듬는 것은 일반 대화처럼 한다. 결과가 나오면 관찰하고, 부족한 부분을 Claude에게 말하면서 rules/나 SKILL.md를 업데이트한다.

대화 예시:
```
(You) /skill recap-mail 고객A
(Claude) [리캡 메일 초안 생성]
(You) "결론 먼저"가 반영되지 않았어. rules/tone.md에 추가해줘.
(Claude) [rules/tone.md 업데이트]
(You) 다시 돌려줘.
(Claude) [개선된 초안]
```

이 과정이 Build → Use → Notice → Fix의 작은 반복이다. 오늘 1~2번의 반복만으로도 70%가 된다.

**관찰 포인트 3가지**

스킬을 돌리고 나면 3가지를 확인한다:
1. **톤**: 내 고객군에 쓸만한가? (임원 회람 vs 개발자 직통)
2. **정확성**: 고객 맥락이 반영됐는가? (MCP 호출이 실제로 됐나)
3. **빠진 것**: 필수 키워드(다음 액션·담당자·링크) 누락 없나

**두 번째 PR — 스킬을 팀 repo에 올리기**

Day 2에서 CLAUDE.md 섹션을 PR로 올렸다. 오늘은 내가 만든 스킬을 팀 repo에 PR로 올린다.

흐름:
```
1. skills/<내이름>/ 폴더 준비 완료
2. git add skills/<내이름>/
3. git commit -m "feat: <내이름> 리캡 메일 스킬 추가"
4. git push
5. Github PR 생성 → 짝꿍 리뷰어 지정
```

PR 제목: `feat: <내이름> 리캡 메일 스킬 v0.5`
내용: "3요소(담당 고객·톤·키워드) 커스터마이징 완료. 체크업 미팅 자료는 2주 프로젝트에서 추가 예정."

---

### 3. CSM 일상 적용

**시나리오 1 — 스킬 첫 실행 후 피드백 반영**

리캡 메일 스킬을 돌렸더니 "결론 먼저"가 안 됐다. Claude에게 "rules/tone.md에 '결론 먼저, 이유·배경은 그 다음' 추가해줘"라고 말하면 된다. 파일을 직접 열어 편집할 필요가 없다.

**시나리오 2 — 짝꿍 피드백으로 다음 반복 결정**

짝꿍이 스킬 결과를 보고 "다음 액션 날짜가 없네"라고 했다. rules/required.md에 "다음 액션 날짜 필수 포함"을 추가한다. 짝꿍 피드백이 스킬 개선 데이터가 된다.

**시나리오 3 — 팀 repo PR이 팀 자산이 되는 순간**

내 스킬이 팀 repo에 merge되면 팀원이 `git pull`로 받아서 바로 쓸 수 있다. 내 30분이 팀 전체의 자산이 되는 순간이다.

---

### 4. 흔한 오해

**오해 1 — "스킬이 70%면 쓰기 어렵다"**

70%는 "보내도 될지 판단하는 수준"이다. 나머지 30%를 고치는 시간이 30%가 아니라 5%로 줄어든다. 이것이 스킬의 가치다. 100%를 목표로 하면 만드는 데 너무 많은 시간이 든다.

**오해 2 — "스킬을 고치려면 파일을 직접 편집해야 한다"**

Claude Code와 대화하면서 "이걸 추가해줘", "이 부분을 이렇게 바꿔줘"라고 말하면 Claude가 파일을 직접 수정한다. 파일을 열어서 편집하는 것보다 빠르다.

**오해 3 — "PR은 완성된 것을 올리는 것이다"**

PR은 "검토 요청"이다. 완성 전에 올려도 된다. 오히려 짝꿍이 리뷰하면서 "이 부분 어떻게 생각해?"라고 대화하는 것이 더 나은 스킬을 만든다.

**Step 1: 베이스 스킬 고르기 + 파일 열기**

리캡 메일 스킬 권장. 운영진 repo에서 복사:
```

```

SKILL.md, rules/, templates/ 3개 파일 열기.

**Step 2: 3요소 수정 (30분)**

- 담당 고객: SKILL.md 입력 섹션을 내 고객군으로
- 톤: rules/tone.md 한 문단 치환 + 예시 문장 추가
- 키워드: rules/banned.md 금지어, rules/required.md 필수 항목

**Step 3: 한 번 실행 + 관찰**
```
/skill <이름>
```
3가지(톤·정확성·빠진 것) 체크. 70% 쓸만하면 합격.

**Step 4: Claude와 대화하며 개선**
```
"결론이 먼저 오지 않았어. rules/tone.md에 추가해줘."
"/skill <이름>  # 다시 실행"
```

**Step 5: 두 번째 PR**
```
git add skills/<내이름>/
git commit -m "feat: <내이름> 리캡 메일 스킬 v0.5"
git push
# Github에서 PR 생성, 짝꿍 리뷰어 지정
```

**막혔을 때**:
- 스킬 호출 안 됨 → SKILL.md 상단에 "이 스킬은 언제 쓰는가" 한 줄 명시
- 톤 반영 안 됨 → SKILL.md에 `rules/tone.md 참조` 문장 추가
- MCP 안 불림 → "이 스킬은 Notion MCP를 사용" 한 줄 추가
