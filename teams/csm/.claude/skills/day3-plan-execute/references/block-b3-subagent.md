## EXPLAIN

### 1. 핵심 한 줄
Subagent는 메인 AI가 여러 보조 AI를 동시에 띄워 각각 다른 작업을 시키고 결과를 합치는 것이다. 속도보다 중요한 것은 "기다리는 시간이 반"으로 줄어든다는 것이다.

---

### 2. 슬라이드 흐름 따라 개념 풀이

**Subagent란 무엇인가**

공식 문서 정의: *"서브에이전트는 도구를 호출하고 작업을 수행할 수 있는 Claude의 독립 인스턴스입니다. 메인 에이전트는 여러 서브에이전트를 생성하고 각각에게 작업을 분배합니다."*

쉽게 말하면: 메인 AI가 여러 일회용 AI를 동시에 띄워서 각자 다른 일을 시키고, 결과를 한 데 모은다. 보조 AI들은 일을 마치면 사라진다.

핵심 특성:
- **일회성**: Subagent는 대화 맥락을 이어받지 않는다. 독립된 일회용 작업자.
- **병렬 처리**: 동시에 여러 작업을 진행한다. 순차 처리의 시간 절약.
- **맥락 격리**: Subagent끼리는 서로를 못 본다. 필요한 맥락은 입력으로 넘겨야 한다.

**순차 vs 병렬 — 리캡 메일 예시**

순차 처리 (기존):
```
고객 히스토리 요약 (30초)
→ 메일 톤 분석 (30초)
→ 리캡 메일 작성 (30초)
= 총 90초
```

Subagent 병렬 처리:
```
고객 히스토리 요약 + 메일 톤 분석 (동시, 30초)
→ 리캡 메일 작성 (30초)
= 총 60초
```

차이: 40초. 하지만 더 중요한 것은 **"내가 기다리는 시간이 반"**이다. 30개 고객 리캡을 만들 때 이 차이가 쌓인다.

**CSM에서 Subagent가 빛나는 곳**

| 작업 | Subagent 활용 방법 |
|-----|-----------------|
| 리캡 메일 생성 | 히스토리 요약 + 다음 액션 초안 + 톤 체크 병렬 |
| 문의 답변 | 과거 답변 검색 + 관련 문서 조회 + 답변 초안 병렬 |
| 체크업 미팅 준비 | 지표 요약 + 지난 아젠다 + 팀 논의 수집 병렬 |

공통점: **여러 곳에서 동시에 끌어와야 답이 완성되는 작업**.

**Subagent 주의점 3가지**

1. **맥락 공유 안 됨**: Subagent는 메인 대화를 못 본다. 필요한 정보는 "입력으로 넘기기". "고객 A의 계약 구조"를 Subagent에게 알려주려면 명시적으로 전달해야 한다.

2. **비용 증폭**: 병렬 호출 = 토큰 합산. 3개 Subagent를 동시에 띄우면 비용도 3배에 가깝다. 꼭 필요한 작업에만 쓴다.

3. **결과 합치기 책임**: 메인 AI가 각 Subagent 결과를 합치는 프롬프트가 필요하다. 그냥 나열하면 읽기 어렵다.

**오늘의 목표**

지금은 "개념만 알고 예제 1번 돌림". 본격적인 활용은 2주 프로젝트에서.

### 3. CSM 일상 적용

**시나리오 1 — QBR 준비 병렬 처리**

Subagent 3개:
- A: SFDC에서 계약·OPP 현황 가져오기
- B: Notion에서 지난 QBR 노트 핵심 추출
- C: Slack에서 이번 주 이 고객 관련 논의 수집

동시에 실행 → 메인 AI가 합쳐서 1페이지 요약

3개를 순차로 하면 3분, 병렬이면 1분.

**시나리오 2 — 15개 고객 동시 리캡**

15개 고객 리캡을 동시에 Subagent 15개로 처리하면 순차 대비 시간이 크게 줄어든다. 매주 목요일 리캡 마감이 있는 CSM에게 의미 있는 시간 절약이다.

**시나리오 3 — 문의 답변 품질 높이기**

Subagent 2개:
- A: 과거 유사 문의 답변 검색
- B: 관련 문서/FAQ 검색

병렬로 돌리고 메인 AI가 결합해서 답변 초안 작성. 순차로 하면 두 검색이 끝날 때까지 기다려야 한다.

---

### 4. 흔한 오해

**오해 1 — "Subagent를 많이 쓸수록 좋다"**

비용이 선형으로 증가한다. 3개 Subagent면 비용도 3배에 가깝다. 정말 병렬로 해야 이득인 작업인지 먼저 판단한다. 단순한 작업은 순차로 하는 게 낫다.

**오해 2 — "Subagent는 서로 대화하면서 협력한다"**

아니다. Subagent는 서로를 모른다. 각자 독립적으로 작업하고 결과를 메인 AI에게 올려준다. 협력이 필요한 구조라면 설계가 달라야 한다.

**오해 3 — "Subagent에게는 맥락을 안 줘도 된다 — AI가 알아서 안다"**

Subagent는 일회용으로 생성되기 때문에 메인 대화의 맥락을 갖지 않는다. "고객 A는 엔터프라이즈" 같은 정보를 Subagent에게 줘야 한다면 명시적으로 입력에 포함시켜야 한다.
