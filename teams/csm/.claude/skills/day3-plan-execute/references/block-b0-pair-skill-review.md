## EXPLAIN

### 1. 핵심 한 줄

어제 만든 내 스킬을 짝꿍이 입력자 입장에서 한 번 호출해보면, 내가 못 본 빈틈이 5분 안에 보인다.

---

### 2. 슬라이드 흐름 따라 개념 풀이

**Day 2에서 한 일 — 복기**

어제 마지막에 본인이 한 사이클을 돌렸다. 자연어 부탁 → SKILL.md 읽기 → Clarify 4가지 → 재부탁. 그리고 마지막에 PR 1개를 올렸다.

지금 그 SKILL.md는 본인 머릿속의 흐름을 그대로 옮긴 상태다. 본인은 "괜찮은 것 같다"고 느끼지만, 그 안에 **본인이 당연하다고 여겨서 안 적은 것**이 반드시 있다.

**짝꿍 점검의 가치**

만든 사람은 자기 결과의 빈틈을 못 본다. 본인이 한 번 더 Clarify를 돌려도, 본인 머릿속 가정 안에서만 다듬어진다. 그래서 짝꿍이 한 번 봐주는 것이 가장 빠르다.

```
[본인 SKILL.md] ── 본인이 다시 봄 → 같은 가정 안에서 다듬어짐
[본인 SKILL.md] ── 짝꿍이 봄 → 다른 가정으로 충돌 → 빈틈 노출
```

이 구조가 오늘 B2에서 배울 **역할 분리(Generator vs Evaluator)** 의 사람 버전이다. 짝꿍이 Evaluator 역할을 한다.

**짝꿍이 해야 하는 것**

짝꿍은 **입력자 입장**에서 본인 스킬을 한 번 호출한다. 본인 짝꿍의 SKILL.md를 받아서 본인이 입력자가 되어 부탁한다.

확인 포인트는 3가지다:

| # | 확인 항목 | 무엇이 보이는가 |
|---|---|---|
| 1 | 입력값 가이드가 명확한가? | "이걸 줘"가 분명한지, 막연한지 |
| 2 | 출력 형태(포맷)가 정해져 있는가? | 불릿 vs 단문 vs 표 — 결과물 모양 예측 가능한지 |
| 3 | 빠진 항목이 있는가? | "이건 당연히 들어가야 하는데 없네"가 있는지 |

본인이 만든 SKILL.md가 어떻게 보이는지를 짝꿍 입장에서 메모로 받는다.

**메모는 B1 입력**

짝꿍 메모 1-2줄이 다음 블록(B1)의 사이클 첫 입력이다. 본인 케이스 #1을 돌릴 때 "짝꿍이 지적한 부분을 어떻게 보완할지"가 이미 손에 잡혀 있어야 한다.

---

### 3. CSM 일상 적용

**예시 — 리캡 메일 스킬**

본인이 어제 만든 SKILL.md가 "고객 미팅 리캡 메일 작성 스킬"이라고 하자.

짝꿍이 받아서 입력자 입장에서 호출한다:

```
짝꿍이 입력하는 것:
"내가 어제 [고객] 미팅을 했어. 리캡 메일 써줘"
```

짝꿍이 결과를 보고 메모할 수 있는 빈틈 예시:

- "참석자 정보가 입력에 없어도 결과가 나오긴 하는데, '담당자 X가 우려한 점'이 임의로 채워져 있어"
- "출력 형태가 매번 다르게 나와. 불릿이었다가 문단이었다가"
- "다음 액션 담당자가 명시되지 않은 채 끝남"

이런 메모가 본인이 못 본 부분이다. 본인은 미팅 맥락을 머릿속으로 알고 있어서 결과만 봤을 때 "괜찮네"라고 느끼지만, 입력만 가지고 결과가 나오는 사람의 입장에서 보면 빈틈이 보인다.

---

### 4. 흔한 오해

**오해 1 — "짝꿍이 본인 업무 도메인을 모르면 점검 못 한다"**

아니다. 본인 도메인을 모르는 짝꿍이 더 빨리 빈틈을 잡는다. 도메인을 알면 본인처럼 머릿속으로 채워서 읽기 때문에 빈틈을 그냥 넘어간다. 도메인을 모르는 짝꿍은 SKILL.md에 적힌 것만으로 판단해야 해서 빠진 항목이 즉시 드러난다.

**오해 2 — "짝꿍이 SKILL.md를 다시 써줘야 한다"**

쓰지 않는다. 메모 1-2줄이면 충분하다. SKILL.md를 다시 쓰는 것은 본인이 B1에서 사이클을 돌리며 한다.

**오해 3 — "지금 SKILL.md를 완벽하게 만들어야 한다"**

아니다. 어제 SKILL.md는 70%다. 짝꿍 점검에서 5분 안에 빈틈 1-2개를 발견하는 것이 목표다. 90%로 가는 길은 B1·B3에서 진행한다.

## EXECUTE

**지금 직접 실행해보세요**

### 1단계 — 짝꿍과 SKILL.md 교환 (2분)

본인의 어제 SKILL.md를 짝꿍에게 보여준다. 짝꿍의 SKILL.md를 본인이 받는다.

화면 공유 / Slack DM / GitHub 링크 어느 방식이든 좋다.

---

### 2단계 — 짝꿍이 입력자로 한 번 호출 (5-7분)

짝꿍은 받은 SKILL.md를 보고, 본인이라면 어떤 입력으로 어떻게 호출할지 자연어로 부탁해본다.

```
짝꿍이 입력하는 형태:
"내가 [짝꿍의 가상 상황]에서 [짝꿍이 만들고 싶은 결과] 만들고 싶어.
이 스킬로 도와줘."
```

결과를 받으면 멈춘다. 결과 자체가 핵심이 아니다. **호출 과정**에서 본 것이 핵심이다.

---

### 3단계 — 빈틈 메모 1-2줄 (3분)

짝꿍이 본인에게 메모를 준다. 형식은 자유, 1-2줄로 충분하다.

```
메모 형식 예시:

"입력에 [항목]이 없어도 결과가 나왔는데, [임의로 채워진 부분]이 있었어"
"출력 형태가 [불릿/단문/표 중 어디였는지] 매번 달랐어"
"[당연히 들어가야 할 것 같은데] 안 들어간 항목이 있어"
```

본인도 짝꿍에게 동일하게 메모를 준다. 서로 1-2줄씩, 양방향.

---

### 4단계 — 짝꿍 메모를 B1 입력으로 (2-3분)

받은 메모를 옆에 두고 다음 블록(B1)으로 간다.

본인 케이스 #1을 돌릴 때 짝꿍이 지적한 부분을 처음부터 의식하면서 자연어 부탁을 한다.

---

**완료 기준**

- [ ] 본인 SKILL.md를 짝꿍에게 보여줌
- [ ] 짝꿍이 입력자로 한 번 호출 시도
- [ ] 짝꿍 메모 1-2줄을 받음
- [ ] 메모를 B1 입력으로 가져갈 준비 완료

---

**관찰 포인트**

짝꿍이 메모로 적은 것이 본인이 머릿속에서 "당연하다"고 여긴 항목인가? 그게 본인의 암묵지다. B1에서 같은 항목이 케이스 #1·#2에서도 빠지면, 본인의 패턴이 확정된다.

이 흐름이 B2의 역할 분리(Generator vs Evaluator)와 같은 구조다. 거기서는 사람 짝꿍이 아니라 Evaluator Subagent가 같은 역할을 한다.
