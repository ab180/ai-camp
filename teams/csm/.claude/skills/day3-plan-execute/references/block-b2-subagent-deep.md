## EXPLAIN

### 핵심 한 줄

Subagent = 메인 AI가 또 다른 독립 AI를 호출해 일을 맡기고, 결과만 돌려받는다. 오늘 아침 B0에서 짝꿍이 한 일(외부 시선으로 본인 스킬 점검)이 사람 버전이고, B2의 역할 분리(Generator vs Evaluator)가 AI 버전이다.

---

### 개념 풀이

**"알바 N명에게 나눠 시킨다"**

내가 직접 모든 고객 상태를 하나씩 확인하는 대신, AI가 알바 5명을 동시에 띄워서 각자 1명씩 맡긴다. 5명이 동시에 작업하고, 나에게는 완성된 요약만 올라온다.

Subagent의 3가지 특성:

| 특성 | 의미 |
|------|------|
| 독립 컨텍스트 | 메인 대화 내용을 이어받지 않는다. 필요한 정보는 명시적으로 전달해야 함 |
| 병렬 처리 | 동시에 N개 작업 가능. 순차 대비 시간 단축 |
| 결과만 반환 | 작업이 끝나면 결과만 메인 AI에게 올린다. 과정은 메인 컨텍스트에 없음 |

---

### 3패턴 (핵심)

#### 패턴 1 — 병렬 검색

여러 작업을 동시에 실행해 시간을 단축한다.

```
[메인 AI]
    ├── Subagent A: 고객 A 상태 점검
    ├── Subagent B: 고객 B 상태 점검
    ├── Subagent C: 고객 C 상태 점검
    ├── Subagent D: 고객 D 상태 점검
    └── Subagent E: 고객 E 상태 점검
             ↓ (동시 실행)
    [메인 AI] 5건 결과 취합
```

**순차 처리**: 30초 × 5건 = 150초
**병렬 처리**: 30초 (5건 동시) = 30초

CSM 사례: "지난주 미팅 5건의 액션 아이템을 한 번에 추출해줘"

---

#### 패턴 2 — 역할 분리 (Generator vs Evaluator)

작성자와 검토자를 분리한다. 만든 사람은 자기 결과를 제대로 보지 못한다.

```
[메인 AI]
    ├── Subagent A (Generator): 리캡 메일 초안 작성
    └── Subagent B (Evaluator): "까다로운 고객" 페르소나로 위 초안 검토
             ↓
    [메인 AI] 검토 의견 받아 수정
```

핵심: Generator가 자기 결과를 검토하면 자신의 맹점을 그대로 넘어간다. Evaluator를 분리하면 다른 시각에서 검토가 가능하다.

CSM 사례: "리캡 메일 초안 작성 → 까다로운 고객 페르소나로 검토"

---

#### 패턴 3 — 컨텍스트 보호

큰 데이터를 Subagent에게 맡기고, 메인 AI는 요약만 받는다. 메인 컨텍스트가 오염되지 않는다.

```
[메인 AI]
    └── Subagent: 회의록 100페이지 분석 + 요약
             ↓ (핵심 3줄만 반환)
    [메인 AI] 요약 3줄로 다음 작업 진행
```

메인 AI가 100페이지를 직접 읽으면 컨텍스트 창이 꽉 찬다. Subagent에게 맡기면 메인은 결과 3줄만 받는다.

CSM 사례: "긴 회의록 압축 정리해서 짝꿍 공유용 요약 생성"

---

### 3패턴 요약표

| 패턴 | 목적 | CSM 대표 케이스 |
|------|------|---------------|
| 병렬 검색 | 시간 단축 | 미팅 5건 동시 액션 추출 |
| 역할 분리 | 품질 향상 | 메일 초안 작성 + 검토자 분리 |
| 컨텍스트 보호 | 컨텍스트 절약 | 긴 회의록 → 핵심 요약만 수신 |

---

### 보너스 — 스킬 체이닝 (Subagent와 다른 축)

**스킬 체이닝 = 여러 작업을 순차 연결**

스킬 A의 출력이 스킬 B의 입력이 되는 흐름. 한 작업을 한 스킬로 끝내려 하지 말고, 작은 스킬 2-3개를 이어붙이면 결과 품질이 올라간다.

```
[입력: 미팅 메모]
    ↓
[스킬 A: 리캡 메일 작성]
    ↓ 출력 = 메일 본문
[스킬 B: 톤·금지어 검토]
    ↓ 출력 = 수정된 본문
[최종: 발송 초안]
```

**Subagent vs 스킬 체이닝 — 축이 다르다**

| | Subagent | 스킬 체이닝 |
|---|---|---|
| 축 | 한 작업 분리 | 여러 작업 순차 연결 |
| 형태 | 한 입력 → 여러 독립 AI에 분산 | 한 입력 → 스킬 A → 스킬 B → 최종 |
| 효과 | 병렬 / 품질 / 컨텍스트 절약 | 단계별 책임 분리 + 재사용 |

**함께 쓰면 위력 배가**: 체인의 한 단계(예: 5건 처리)를 Subagent 병렬 패턴으로 돌리면 전체 흐름이 빨라진다.

**CSM 체이닝 후보 3**:
- 리캡 메일 작성 + 톤·금지어 검토
- 미팅 노트 요약 + 다음 액션 추출
- 문의 유형 분류 + 유형별 답변 호출

**오늘은 개념만**. 실전 체이닝은 Day 4 Compound 사이클에서 다룬다 — 검증 결과를 다음 스킬 입력으로 자동 흘리는 구조.

### CSM 적용 시나리오

**시나리오 1 — 병렬 패턴 활용 (미팅 리캡 마감일)**

매주 금요일 오후 5건의 미팅 노트가 쌓여 있다. 하나씩 읽고 액션 추출하면 1시간. Subagent 5개로 동시 처리하면 15분.

**시나리오 2 — 역할 분리 활용 (갱신 리스크 고객 메일)**

갱신을 앞두고 불만이 있는 고객에게 메일을 쓸 때, 내가 쓴 초안을 나 스스로 검토하면 놓치는 부분이 있다. "까다로운 고객 페르소나로 이 메일을 보면 어떤 반응이 나오는지 검토해줘"라고 Evaluator를 붙이면 다른 시각의 피드백이 나온다.

**시나리오 3 — 컨텍스트 보호 활용 (분기 리뷰 자료 준비)**

QBR 자료 준비를 위해 지난 3개월 미팅 노트를 전부 읽어야 한다. 메인 AI에게 직접 주면 컨텍스트가 꽉 찬다. Subagent에게 "3개월치 노트에서 핵심 이슈 5개와 고객 감정 변화 추이만 추출해줘"라고 시키고, 결과만 받는다.

---

### 흔한 오해

**오해 1 — "Subagent는 무조건 빠르다"**

시간은 단축되지만 토큰 비용은 비례해서 증가한다. Subagent 5개 = 비용 약 5배. 짧고 단순한 작업에는 오히려 손해다. 정말 병렬로 해야 이득인지 먼저 판단한다.

**오해 2 — "Subagent가 알아서 검증한다"**

아니다. Subagent도 환각 위험이 동일하다. 메인 AI 또는 사람이 결과를 검증해야 한다. "Subagent가 했으니 믿어도 된다"는 착각이 가장 위험하다.

**오해 3 — "Subagent = 스킬과 같다"**

다르다. 스킬(SKILL.md)은 절차서 — AI가 어떻게 행동할지 정의한 문서다. Subagent는 실행자 — 그 절차를 수행하는 독립된 AI 인스턴스다. 스킬 안에서 Subagent를 호출할 수도 있다.

**오해 4 — "Subagent는 메인 대화를 본다"**

보지 못한다. Subagent는 독립 컨텍스트로 시작한다. "지금까지 우리가 얘기한 내용 기반으로"라고 해도 Subagent는 모른다. 필요한 맥락은 명시적으로 입력에 포함해야 한다.

**1단계 — Generator: 본인 스킬로 결과 1건 생성 (5분)**

Day 2에 만든 SKILL.md를 실행해 결과물 1건을 만든다.
(예: 리캡 메일 초안, 고객 상태 요약, 미팅 액션 리스트 등)

결과를 그대로 두고 다음 단계로 넘어간다.

---

**2단계 — Evaluator: 까다로운 고객 페르소나로 검토 (5분)**

Claude에게 자연어로 요청한다:

```
위에서 만든 [결과물 이름]을 검토해줘.
검토 기준: [고객사명] 담당자가 갱신을 고민 중이고
지난 분기 지표 달성이 아쉬웠던 상황이야.
이 메일/요약을 받으면 어떤 반응이 나올지,
빠진 게 있거나 오해를 살 문장이 있으면 짚어줘.
```

---

**3단계 — 검토 결과 메모 (3분)**

검토 의견을 받은 후, 메모장에 1-2줄로 적는다:

- "어떤 부분이 부족하다고 나왔나?"
- "내가 놓쳤던 건 무엇인가?"

이 메모가 Day 4 "역할 분리 심화" 실습의 인풋이 된다.

---

**관찰 포인트**

Generator(나 또는 AI)가 만든 결과와 Evaluator의 검토 의견 사이에 gap이 있는가? 그 gap이 역할 분리의 가치다.
