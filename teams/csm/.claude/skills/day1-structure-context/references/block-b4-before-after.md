## EXPLAIN

### 1. 한 문장 요약
같은 한 줄 요청도 CLAUDE.md 유무에 따라 결과의 "두께"가 달라지는 것은, AI가 더 똑똑해진 게 아니라 이미 알고 있는 맥락의 양이 달라졌기 때문이다.

---

### 2. 왜 필요한가 (Why)

**"쓸 만한데 내 것이 아니다" — 이게 문제의 핵심**

CLAUDE.md 없이 AI 도움을 받으면 이런 경험을 하게 된다.

```
요청: "고객 X 리캡 메일 써줘"

결과:
Hi [Customer],
Thanks for the great meeting.
Here's a recap of our discussion...
Next step: follow-up call.
```

읽어보면 틀린 곳이 없다. 하지만 어딘가 "내가 쓴 것 같지 않다"는 느낌이 든다. 수정을 시작하는데, 이름도 바꾸고, 톤도 다듬고, 내용도 구체화하다 보면 — 원래 초안의 30~40%는 지워진다. 결과적으로 시간이 얼마나 줄었는지 애매해진다.

이 문제의 원인은 AI의 능력이 아니다. 맥락의 부재다.

AI에게 "리캡 메일 써줘"라고 말할 때, AI가 모르는 것들:
- 이 고객이 엔터프라이즈인지, 스타트업인지
- 담당자 이름이 무엇인지
- 우리 팀이 어떤 톤으로 쓰는지
- 어떤 표현을 쓰면 안 되는지
- 이번 미팅에서 어떤 합의가 있었는지

AI는 이것들을 알 수 없으니, 어느 고객에게나 통하는 일반 템플릿으로 채운다.

**CLAUDE.md가 이 공백을 채운다**

CLAUDE.md에 기본값이 적혀 있으면, 같은 한 줄 요청에 대해 AI는 이미 알고 있는 상태에서 시작한다. 결과가 달라지는 건 AI가 더 똑똑해진 게 아니라, "처음 만난 상태"가 사라졌기 때문이다.

---

### 3. 본질 (What)

**맥락의 두께**

Before/After를 표면적으로 보면 "이름이 호명된다", "톤이 맞다"처럼 항목이 달라진 것처럼 보인다. 하지만 더 깊은 변화는 "맥락의 두께"다.

```
[CLAUDE.md 없을 때 — 맥락 0겹]
  AI가 가진 것: 요청 한 줄
  AI가 모르는 것: 고객 정보, 톤, 금지어, 워크플로우

  → AI는 공백을 일반 패턴으로 채운다

[CLAUDE.md 있을 때 — 맥락 N겹]
  AI가 가진 것: 요청 한 줄 + 고객 유형 + 담당자 이름 + 톤 + 금지어 + 워크플로우
  AI가 모르는 것: 이번 미팅의 구체적 내용만

  → AI는 알고 있는 기반 위에서 빠진 부분만 추정한다
```

"맥락의 두께"가 다르면, 같은 요청이 완전히 다른 작업이 된다.

**비유: 의사의 진료**

```
첫 방문 환자 (CLAUDE.md 없을 때)
  의사: "어디가 불편하세요?"
  환자: "어깨가 아파요"
  의사: 일반적인 어깨 통증 처치 → 맞을 수도 있고 아닐 수도 있다

단골 환자 (CLAUDE.md 있을 때)
  의사: 차트에 3년치 기록이 있다
  환자: "어깨가 또 아파요"
  의사: "지난번 MRI에서 이 부위였죠? 이번엔 이렇게 해봅시다"
  → 즉시 맥락 위에서 진료
```

CLAUDE.md는 AI의 차트다. 한 번 써두면 이후 모든 대화에서 "단골" 처우를 받는다.

**Before/After 실제 비교**

```
[CLAUDE.md 없을 때]
프롬프트: "고객 X 리캡 메일 써줘"

결과:
  Hi [Customer],
  Thanks for the great meeting.
  Here's a recap of our discussion...
  Next step: follow-up call.

문제점:
  - 고객 이름 없음 → 수동으로 바꿔야 함
  - 일반 비즈니스 영어 → 우리 팀 톤과 다름
  - 구체적 합의사항 없음 → 내가 추가해야 함
  - 다음 액션 날짜 없음 → 고객이 "언제?"를 다시 물음
  수정 시간: 10~15분

[CLAUDE.md 있을 때]
같은 프롬프트: "고객 X 리캡 메일 써줘"

결과:
  이에나님,
  어제 대시보드 세팅 미팅 빠르게 진행됐습니다.
  합의한 이벤트 목록은 이번 주 목요일까지 공유드리겠습니다.
  다음 체크업은 2주 뒤 화요일 오후 2시로 잡아도 될까요?

달라진 점:
  - 이름이 실제로 호명됨
  - 짧고 따뜻한 한국어 톤
  - 오늘 합의한 것이 구체적으로 명시됨
  - 다음 액션에 기한이 붙어 있음
  수정 시간: 2~3분
```

수정 시간이 10분 줄어든 게 아니다. 판단의 에너지가 줄었다. "이거 맞나 틀리나"를 매번 고민하는 대신, "이건 보내면 되겠다" 수준으로 바로 판단할 수 있다.

### 4. 어떻게 작동하나 (How)

**CLAUDE.md가 before/after 차이를 만드는 메커니즘**

```
Claude Code 세션 시작
          ↓
CLAUDE.md가 있는 폴더에서 시작했나? → YES
          ↓
Claude: CLAUDE.md 내용을 컨텍스트 첫 줄에 로딩
          ↓
사용자: "고객 X 리캡 메일 써줘"
          ↓
Claude: 이미 알고 있는 것 (CLAUDE.md 내용) + 요청 내용을 합쳐서 처리
          ↓
출력: 기본값이 반영된 결과
```

중요한 것은 사용자가 아무것도 설명하지 않았다는 점이다. "우리 팀은 짧고 따뜻한 톤이에요"를 매번 붙여넣지 않아도 된다.

**차이가 커지는 시점**

처음에는 before/after 차이가 미미하게 느껴질 수 있다. CLAUDE.md가 아직 얇기 때문이다. 하지만 쓰면서 채울수록 차이가 커진다.

```
CLAUDE.md 0.5판 (오늘)
  → 이름·톤 정도 반영 (수정 시간 30% 감소)

CLAUDE.md 1.0판 (한 달 후)
  → 담당 고객별 특성 + 금지어 + 워크플로우까지 (수정 시간 60% 감소)

CLAUDE.md 2.0판 (3개월 후)
  → 반복 지적이 반영된 정교한 규칙 (수정 시간 80% 감소 + 보내도 되는 판단만)
```

오늘의 before/after는 작다. 하지만 이게 복리로 쌓인다.

### 5. 흔한 오해 (Common Misconceptions)

**오해 1: "CLAUDE.md가 있으면 수정이 필요 없다"**

아니다. CLAUDE.md는 기본값이지 완성본 생성기가 아니다. 이번 미팅의 구체적 내용은 내가 전달해야 한다. CLAUDE.md는 "이미 알아야 할 것"을 미리 담아두는 곳이고, "이번에 일어난 일"은 그때그때 요청에 담아야 한다.

수정이 0이 되는 게 목표가 아니다. "한 번 읽고 보내도 되겠다"는 판단이 쉬워지는 것이 목표다.

**오해 2: "before 결과도 충분히 쓸 만하다"**

그렇게 느껴지는 이유가 있다. 읽는 사람이 만든 사람이기 때문이다. 내 머릿속의 맥락이 자동으로 채워져서 그럴듯하게 읽힌다. 하지만 실제로 고객이 받았을 때, "고객 A"가 "[Customer]"로 적혀 있다면, 또는 내용이 어느 고객에게나 통하는 일반적인 말이라면, 메일의 온기가 사라진다.

**오해 3: "after 결과는 내가 안 쓴 것 같다"**

처음에 그렇게 느끼는 사람이 많다. CLAUDE.md에 내가 직접 적은 기본값으로 AI가 채운 결과인데도 "내가 안 쓴 것 같다"고 느끼는 이유는 익숙하지 않아서다. 한 달쯤 쓰고 나면, after 결과가 "내 스타일"로 느껴지게 된다.

### 6. CSM 일상 적용

**시나리오 1: 주간 리캡 메일 5건**

월요일 오전, 지난 주 미팅 5건의 리캡 메일을 써야 한다.

CLAUDE.md 없이: 각 메일마다 고객 이름, 톤, 다음 액션을 신경 쓰며 작성 → 건당 20분 = 1시간 40분

CLAUDE.md 있고 각 고객의 미팅 노트를 넣으면: "고객 A 리캡 메일" → before 검토 2분 + 수정 2분 = 건당 4분 = 20분

**시나리오 2: 갑작스러운 Slack 문의**

Slack으로 고객이 긴급 질문을 보냈다. 바로 답변 초안을 뽑아야 하는데, CLAUDE.md 없이는 이 고객이 어떤 맥락인지 설명을 한 단락 붙여야 한다. CLAUDE.md에 고객군 특성이 있으면 "고객 B 이 질문에 답변해줘"만 해도 맥락이 담긴 초안이 나온다.

**시나리오 3: 체크업 미팅 준비**

미팅 30분 전, CLAUDE.md에 담당 고객 리스트와 워크플로우가 있으면 "고객 C 체크업 준비 자료 만들어줘"로 시작할 수 있다. 없으면 "이 고객은 어떤 계약 구조이고, 어디 단계에 있고..."를 설명하는 데 준비 시간의 절반이 간다.

### 지금 직접 비교해보세요

이 실습은 오늘 만든 CLAUDE.md의 효과를 눈으로 확인하는 과정입니다.

**준비**: Claude Code가 열려 있고, B3에서 6개 섹션을 채운 CLAUDE.md가 있는 상태.

**단계 1: 요청 입력**

Claude Code에서 아래를 입력합니다:
```
고객 [실제 담당 고객 이름] 어제 미팅 후 리캡 메일 한 단락으로 써줘.
어제 주요 논의 내용: [한두 문장으로 간단히]
```

**단계 2: 결과 확인 — 3가지 포인트**

나온 결과를 보면서 체크합니다:
- 고객 이름이 실제로 호명되는가? (이름이 없거나 "[Customer]"이면 담당 고객 섹션 재확인)
- CLAUDE.md에 적은 톤이 반영되는가? (딱딱하거나 너무 격식체이면 톤앤매너 섹션 재확인)
- CLAUDE.md에 적은 금지어가 빠지는가? (금지어가 여전히 나오면 금지어 섹션에 이유도 함께 적혀 있는지 재확인)

**단계 3: 하나만 수정해보기**

결과에서 가장 마음에 안 드는 부분 1개를 찾아서 CLAUDE.md를 수정한 뒤, 같은 요청을 다시 넣어보세요. 수정 후 결과가 달라졌나요?

**막혔을 때**:
- 결과가 이전과 별로 달라 보이지 않으면 → CLAUDE.md에 6개 섹션이 실제로 채워져 있는지 확인
- 이름이 안 나오면 → "담당 고객" 섹션에 이름이 명시되어 있는지 확인
- 전혀 모르겠으면 → CLAUDE.md 파일 자체를 Claude에게 보여주고 "어디가 문제야?"라고 물어보기
