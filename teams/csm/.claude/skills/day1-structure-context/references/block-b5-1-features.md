## EXPLAIN

### 1. 한 문장 요약
Claude Code의 7가지 기능은 각각 다른 문제를 해결하며, 그 중 CLAUDE.md만이 "항상 켜진 상태"로 작동한다 — 나머지 6가지는 필요할 때만 꺼내 쓰는 도구다.

---

### 2. 왜 필요한가 (Why)

**"AI 도구는 여러 개인데, 어디서 시작해야 할지 모르겠다"**

Claude Code를 처음 접하면 기능이 너무 많아 보인다. Skill, MCP, Subagent, Hook, Plugin... 다 배워야 하나? 어느 것부터 써야 하나?

이 혼란을 줄이는 가장 좋은 방법은 "각 기능이 어떤 문제를 해결하는지"를 층위로 이해하는 것이다.

```
층위 1: "AI가 매번 내 기본값을 모른다" → CLAUDE.md
층위 2: "반복 작업을 매번 새로 설명해야 한다" → Skill
층위 3: "AI가 내 업무 시스템의 데이터를 못 읽는다" → MCP
층위 4: "비슷한 작업을 여러 고객에게 동시에 하려면 느리다" → Subagent
층위 5: "팀과 내 설정을 공유하고 싶다" → Plugin
```

각 층위는 독립적이지 않다. CLAUDE.md가 쌓이면 Skill이 더 잘 동작하고, Skill이 있으면 MCP와 연결했을 때 자동화 수준이 높아진다. 나중에 배울 기능들이 Day 1의 CLAUDE.md 위에 쌓이는 구조다.

---

### 3. 본질 (What)

**7개 기능의 정확한 역할**

| 기능 | 한 줄 정의 | 언제 쓰나 | 이번 캠프 |
|------|----------|---------|---------|
| **CLAUDE.md** | 매 세션 자동으로 읽히는 기본값 파일 | 항상 (켜져 있음) | Day 1 |
| **Skill** | 슬래시 명령으로 불러쓰는 반복 레시피 | 이 작업에만 불러씀 | Day 2~3 |
| **MCP** | 외부 시스템을 AI에 연결하는 표준 포트 | 연결된 도구가 필요할 때 | Day 2 |
| **Subagent** | 독립된 작업자를 병렬로 띄우기 | 독립된 작업이 2개 이상일 때 | Day 3 |
| **Agent Teams** | 여러 역할의 에이전트가 협업 | (실험적, 심화) | - |
| **Hook** | 이벤트 발생 시 자동 실행 | 파일 저장·세션 시작 등 트리거 | - |
| **Plugin** | 팀이 만든 스킬 패키지 배포 | 팀에 공유할 때 | Day 4 |

**각 기능의 비유**

```
CLAUDE.md  = 사무실 인계서 — 새 직원이 첫날 읽는 것. 한 번 쓰면 계속 효과.
Skill      = 업무 매뉴얼의 단계별 지침 — 필요할 때 꺼내 보는 절차서.
MCP        = USB-C 포트 — 어떤 기기든 같은 포트로 연결.
Subagent   = 팀의 멤버들 — 과장이 여러 사원에게 동시에 과제를 나눠줌.
Plugin     = 회사 표준 키트 — 팀원이라면 모두 같은 도구를 쓸 수 있게 패키지화.
```

**왜 CLAUDE.md가 가장 중요한가**

7개 중 CLAUDE.md만이 "항상 로딩" 범주에 속한다. 나머지 6개는 "필요할 때" 호출하거나 설정해야 한다.

```
오늘 작업하는 내 모든 대화에 영향을 미치는 것: CLAUDE.md만
특정 작업에만 영향을 미치는 것: Skill, MCP, Subagent...
```

비유하자면, CLAUDE.md는 "공기"고 나머지는 "도구"다. 공기가 없으면 도구가 있어도 제대로 작동하지 않는다.

**기능 간 관계**

```
CLAUDE.md (기반)
    ↓ 기본값 제공
  Skill (절차 레시피)
    ↓ 반복 작업 자동화
    + MCP (외부 데이터 연결)
    ↓ 실시간 데이터 포함
    + Subagent (병렬 처리)
    ↓ 여러 고객/작업 동시 처리
  Plugin (팀 공유)
    ↓ 개인 설정 → 팀 자산
```

이 구조를 이해하면, "무엇부터 해야 하는가"가 명확해진다: CLAUDE.md가 기반이다.

---

### 4. 어떻게 작동하나 (How)

**각 기능을 켜는 방법**

| 기능 | 켜는 방법 | 어디서 |
|------|---------|-------|
| CLAUDE.md | 파일이 있으면 자동 | Claude Code 세션 시작 시 |
| Skill | `/스킬이름` 입력 | Claude Code 대화창 |
| MCP | Settings → Integrations에서 연결 | claude.ai 웹 설정 |
| Subagent | Claude가 자동으로 판단 또는 "병렬로" 요청 | 프롬프트 |
| Plugin | 팀 repo에서 설치 | 설정 |

**이번 캠프 4일 커리큘럼과 기능 연결**

```
Day 1: CLAUDE.md 만들기 — 기반 쌓기
Day 2: MCP 연결 + Skill 첫 제작 — 데이터 연결 + 반복 레시피
Day 3: Skill 다듬기 + Subagent 맛보기 — 정교화 + 병렬 처리
Day 4: 팀 공유 + 2주 프로젝트 안내 — 팀 자산화
```

오늘이 Day 1이므로, 7개 기능 중 나머지 6개는 이후 3일에 걸쳐 하나씩 배운다. 지금은 CLAUDE.md 하나만 잘 만드는 것이 전부다.

---

### 5. 흔한 오해 (Common Misconceptions)

**오해 1: "7개를 다 배워야 AI를 잘 쓸 수 있다"**

아니다. CLAUDE.md 하나만 잘 써도 일상 업무에서 체감 효과가 크다. 7개를 모두 쓰는 것이 목표가 아니라, 내 업무 맥락에서 필요한 것만 골라 쓰는 것이 목표다.

**오해 2: "Skill이 있으면 CLAUDE.md가 필요 없다"**

반대다. Skill은 CLAUDE.md의 기본값을 기반으로 더 잘 동작한다. CLAUDE.md에 톤·금지어·워크플로우가 없는 상태에서 Skill을 만들면, Skill의 결과물도 일반적인 수준에 머문다.

**오해 3: "MCP를 연결하면 CLAUDE.md가 자동으로 채워진다"**

MCP는 실시간 외부 데이터를 가져오는 역할이고, CLAUDE.md는 변하지 않는 기본값을 담는 역할이다. 둘은 해결하는 문제가 다르다. MCP로 오늘의 Slack 대화를 가져올 수 있지만, "우리 팀 톤은 짧고 따뜻하게"는 CLAUDE.md에 직접 적어야 한다.

---

### 6. CSM 일상 적용

**시나리오 1: 오늘은 이것 하나만**

4일 캠프를 마치고 다음 주 업무에 돌아갔을 때, CLAUDE.md 하나만 있어도 리캡 메일 초안의 품질이 달라진다. Skill, MCP, Subagent는 이후 자연스럽게 채워갈 수 있다. 기능을 다 배우겠다는 욕심보다 "CLAUDE.md 하나를 내 것으로 만들겠다"가 현실적인 첫 목표다.

**시나리오 2: 3개월 뒤 자연스러운 확장**

CLAUDE.md가 잘 자리잡으면, "이 작업을 반복하고 있는데 Skill로 만들어볼까"가 자연스럽게 생긴다. MCP도 "Notion에 있는 미팅 노트를 매번 붙여넣는 게 번거롭다"고 느낄 때 연결 동기가 생긴다. 기능은 필요를 느낄 때 확장하면 된다.

**시나리오 3: 팀에 전파할 때**

내 CLAUDE.md와 Skill이 잘 작동하면, 팀에 공유하는 것이 Plugin 단계다. "이거 나한테 잘 맞더라"가 먼저고, "팀에도 공유해보자"가 다음이다.
