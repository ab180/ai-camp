## EXPLAIN

### 1. 한 문장 요약
좋은 CLAUDE.md는 길수록 좋은 게 아니라, 짧고 이유가 명확하고 팀이 함께 편집할 수 있어야 한다 — 길어졌다면 그건 추가할 자리가 아니라 덜어낼 신호다.

---

### 2. 왜 필요한가 (Why)

**"많이 넣을수록 좋을 것 같다"는 함정**

CLAUDE.md를 처음 만들면 자연스럽게 이런 생각이 든다. "지금 생각나는 것 다 넣어두자. 나중에 필요하겠지." 그래서 몇 달이 지나면 파일이 500줄이 넘어간다. 그리고 이상한 일이 벌어진다.

AI의 결과물이 오히려 이상해진다.

이유가 있다. AI는 CLAUDE.md를 대화의 첫 맥락으로 읽지만, 파일이 길어질수록 앞부분을 더 중요하게 처리하고 뒷부분을 희석한다. 500줄짜리 CLAUDE.md에서 400번째 줄에 있는 금지어는 거의 무시된다. 내가 열심히 적었는데 AI가 안 읽는 셈이다.

게다가 사람도 유지하기 어렵다. "이 규칙이 왜 있지?" "이 고객사 규칙은 지금도 유효한가?" 읽을 때마다 판단이 필요해진다.

**좋은 CLAUDE.md의 기준**

```
나쁜 CLAUDE.md:
  - 길다 (500줄+)
  - 이유 없는 규칙이 많다
  - 언제 적었는지 모르는 항목이 있다
  - 혼자만 편집한다

좋은 CLAUDE.md:
  - 짧다 (200줄 이내)
  - 모든 규칙에 이유가 있다
  - 변하지 않는 것만 담긴다
  - 팀이 함께 쓸 수 있다
```

이 기준은 Anthropic 공식 가이드에서도 강조된다: "복잡해지고 있다면 더 넣을 자리가 아니라 덜어낼 자리를 찾는 것이 신호다."

---

### 3. 본질 (What)

**5가지 원칙의 의미**

**원칙 1: 짧게 유지 (200줄 이내)**

단순히 AI 성능 때문만이 아니다. 200줄이 넘는 순간 "내가 이걸 진짜 다 읽어봤나?"가 의심되기 시작한다. CLAUDE.md는 완성본이 아니라 살아있는 문서다. 내가 읽고 판단할 수 있는 길이여야 유지된다.

길어지기 시작하면 어디를 덜어낼지 방법이 있다:
```
"자주 안 쓰는 규칙" → .claude/rules/ 폴더로 이동
"변하는 정보" → Notion·Slack에 두고 MCP로 연결
"일회성 고객 특성" → 개별 대화에서 붙여넣기로
```

**원칙 2: CLAUDE.md는 지도 역할, 내용은 밖으로**

비유하면 CLAUDE.md는 건물 안내도다. "3층에 회의실 있음"이라고 적혀 있지, 회의실 내부 사진이 들어 있지 않다.

```
CLAUDE.md (안내도):
  "리캡 메일 규칙은 .claude/rules/recap-mail.md 참고"
  "담당 고객 상세 정보는 docs/clients.md 참고"

실제 내용은 각 파일에:
  .claude/rules/recap-mail.md → 리캡 메일 세부 규칙
  docs/clients.md → 고객별 상세 특성
```

CLAUDE.md가 길어지면 이 구조를 활용하는 신호다.

**원칙 3: 고정 vs 변하는 정보 분리**

CLAUDE.md에 담아야 할 것:
- 팀 톤앤매너 (잘 변하지 않음)
- 금지어 + 이유 (잘 변하지 않음)
- 워크플로우 기본 구조 (잘 변하지 않음)

CLAUDE.md에 담으면 안 되는 것:
- "이번 분기 OKR" → 분기마다 변한다
- "현재 진행 중인 고객 이슈" → 매일 변한다
- "이번 주 미팅 일정" → 매주 변한다

변하는 정보를 CLAUDE.md에 넣으면, 업데이트를 못 한 순간부터 AI가 오래된 기준으로 답하기 시작한다.

**원칙 4: 금지어에 반드시 "왜"**

이유는 B3에서도 강조했지만, best practices에서 한 번 더:

```
이렇게 적으면 AI가 어긴다:
  "반드시"라는 단어 사용 금지

이렇게 적으면 AI가 지킨다:
  "반드시" 금지 — 이유: 기대치를 과도하게 형성해 고객 실망 유발 가능
```

이유가 없으면 AI는 "이 규칙이 맥락상 필요한지 아닌지"를 스스로 판단하려 한다. 그 판단이 틀릴 때 금지어를 써버린다. 이유가 있으면 AI는 "이 이유가 지금 상황에 적용되니까 쓰면 안 되겠다"고 일관되게 판단한다.

**원칙 5: 팀이 함께 편집 가능한 구조**

개인 선호와 팀 규칙이 섞이면 문제가 생긴다.

```
섞이면 안 좋은 예:
  "나는 이모지 쓰는 거 싫어 (개인 선호)"
  "우리 팀은 snake_case 사용 (팀 규칙)"
  → 같은 CLAUDE.md에 있으면 팀원이 clone했을 때 혼란

분리된 예:
  ~/.claude/CLAUDE.md (User 레벨):
    "나는 이모지 쓰는 거 싫어"
  프로젝트/CLAUDE.md (Project 레벨):
    "우리 팀은 snake_case 사용"
```

계층 구조:
```
~/.claude/CLAUDE.md           ← User 레벨: 내 모든 프로젝트에 적용
프로젝트/CLAUDE.md             ← Project 레벨: 이 프로젝트(팀)에만 적용
프로젝트/.claude/rules/        ← 주제별 세부 규칙
프로젝트/docs/                 ← 실제 비즈니스 지식
```

### 4. 어떻게 작동하나 (How)

**5가지 피해야 할 패턴과 대처법**

```
패턴 1: 500줄+ CLAUDE.md
  증상: "리캡 써줘" 하면 엉뚱한 규칙이 적용됨
  원인: AI가 긴 파일에서 앞부분만 집중해서 읽음
  처방: 자주 안 쓰는 규칙을 .claude/rules/로 이동. CLAUDE.md는 포인팅만.

패턴 2: 변하는 정보를 CLAUDE.md에 담기
  증상: 다음 분기가 됐는데 AI가 지난 분기 목표로 답함
  원인: "이번 분기 OKR" 같은 내용이 CLAUDE.md에 있고 업데이트 안 됨
  처방: 변하는 정보는 Notion·Slack에 두고 MCP로 연결. CLAUDE.md는 고정값만.

패턴 3: 이유 없는 금지어
  증상: CLAUDE.md에 "'반드시' 쓰지 마"라고 적었는데 AI가 여전히 씀
  원인: 이유가 없으면 AI가 스스로 판단해서 쓰기로 결정
  처방: "이유: 기대치 과다 형성 우려"처럼 금지 근거 추가.

패턴 4: 개인 선호와 팀 규칙 섞기
  증상: 팀원이 내 CLAUDE.md를 clone했을 때 내 개인 스타일이 팀에 강요됨
  원인: User 레벨과 Project 레벨이 분리 안 됨
  처방: 개인 스타일은 ~/.claude/CLAUDE.md에, 팀 규칙은 프로젝트 CLAUDE.md에.

패턴 5: 안 쓰는 규칙 누적
  증상: 몇 달 뒤 옛 고객사 규칙이 아직 있고 AI가 그걸 따름
  원인: 정기 검토를 안 함
  처방: 3개월마다 "이 중 지금 안 쓰는 항목 정리해줘" 요청.
```

**CLAUDE.md 자가진단 체크리스트**

오늘 만든 CLAUDE.md를 이 5가지로 점검해볼 수 있다:

```
□ 200줄 이내인가?
□ 변하는 정보(분기 OKR, 현재 이슈)가 섞여 있지 않은가?
□ 금지어에 이유가 같이 적혀 있는가?
□ 개인 선호와 팀 규칙이 분리되어 있는가?
□ 읽으면서 "이건 왜 있지?"라는 항목이 없는가?
```

### 5. 흔한 오해 (Common Misconceptions)

**오해 1: "많이 넣을수록 AI가 더 잘 안다"**

역효과가 난다. AI는 긴 파일에서 중요도를 자체 판단한다. 내가 400번째 줄에 적은 중요한 규칙이 AI에게는 희미하게 처리된다. 짧고 핵심만 있는 CLAUDE.md가 긴 것보다 실제로 더 잘 작동한다.

**오해 2: "규칙은 많을수록 더 세밀하게 통제된다"**

규칙이 많아지면 오히려 충돌이 생긴다. "메일은 짧게"와 "모든 맥락을 포함해서"가 같이 있으면 AI는 둘 사이에서 임의로 결정한다. 규칙 간 우선순위나 충돌이 생기지 않도록, 핵심적인 것만 남기는 것이 더 예측 가능한 결과를 만든다.

**오해 3: "한 번 만들면 그대로 써야 한다"**

CLAUDE.md는 계속 편집하는 문서다. 쓰다가 "이 규칙이 맞지 않다"는 느낌이 오면 바로 고쳐야 한다. "완성된 CLAUDE.md"는 없다. 항상 조금 더 정교해지는 과정에 있는 문서다.

### 6. CSM 일상 적용

**시나리오 1: 3개월 후 CLAUDE.md 검토**

처음 만든 CLAUDE.md를 3개월 후에 열어보면 이런 항목들이 눈에 띈다. "이 고객사 규칙은 계약이 끝났는데 아직 있네", "이 금지어는 이제 우리 팀에서 쓰지 않는데". 정기적으로 Claude Code에서 "내 CLAUDE.md에서 지금 안 쓰는 항목이 있는지 확인해줘"라고 요청하면 청소할 항목을 제안받을 수 있다.

**시나리오 2: 팀원이 내 CLAUDE.md를 쓰려 할 때**

내 CLAUDE.md를 팀원에게 공유하면, User 레벨(내 개인 스타일)과 Project 레벨(팀 규칙)이 섞여 있을 가능성이 높다. 팀 공유 전에 개인 선호는 따로 분리해서 프로젝트 CLAUDE.md에는 팀이 동의한 내용만 남겨두는 것이 좋다.

**시나리오 3: 새 고객을 담당할 때**

담당 고객이 바뀌면 CLAUDE.md를 바로 업데이트해야 한다. 예전 고객 정보가 남아 있으면 AI가 잘못된 맥락으로 메일 초안을 만들 수 있다. "변하는 정보"인 개별 고객 상세는 별도 파일에 두고 CLAUDE.md에서 포인팅하는 구조가 이럴 때 유용하다.
