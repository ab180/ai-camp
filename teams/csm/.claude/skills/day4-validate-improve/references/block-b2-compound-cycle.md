## EXPLAIN

### 1. 핵심 한 줄
Build→Use→Notice→Fix 사이클이 반복될수록 스킬이 나아지고, 6축 자가진단은 지금 내가 어디에 서 있는지를 보는 도구다.

---

### 2. 슬라이드 흐름 따라 개념 풀이

**Compound Engineering 사이클**

"Compound"는 복리를 뜻한다. 스킬이 개선되면 그 스킬로 만든 결과가 나아지고, 나아진 결과가 다시 스킬 개선 재료가 된다. 이 순환이 복리처럼 쌓인다.

4단계 사이클:
```
Build  → 스킬·CLAUDE.md·rules 만들기
  ↓
Use    → 실제 업무에 사용
  ↓
Notice → 3번 이상 반복되는 결함 발견 ("이 표현이 계속 이상하게 나온다")
  ↓
Fix    → 결함을 rules/CLAUDE.md에 반영
  ↓
다시 Build...
```

**"3번 반복되는 결함 = 스킬/rules 자리"**

한 번 나온 문제는 우연일 수 있다. 두 번이면 의심. 세 번이면 **구조적 결함**이다. 세 번 이상 같은 문제가 나오면:
- "이건 매번 내가 고치는 게 아니라 스킬에 박아야 한다"
- rules/banned.md 또는 rules/required.md에 추가

이것이 Notice → Fix 단계다.

**Compound는 자동화하지 않으면 깨진다 — 왜 wrap이 필요한가**

Compound 사이클(Build→Use→Notice→Fix)에서 가장 무너지기 쉬운 단계는 **Notice**다. 사람이 매번 "오늘 이 패턴이 반복됐다"를 의식적으로 적어야 한다. 일주일 일하면 잊어버린다.

자동화/구조화 없이 Compound가 굴러가는 사람은 거의 없다. 캠프에서 70%가 다음 주 회고 안 적는다. 첫 주는 의지로 가능한데 둘째 주부터 안 된다.

→ Compound가 진짜 굴러가려면 **Notice를 자동화**해야 한다. 그 자동화의 구체 예시가 `/wrap` 스킬.

---

**wrap = Compound 자동화의 구체 예시 #1**

`/wrap`은 세션 끝에 한 번 호출하면 다음 5가지를 자동으로 한다:

```
세션 끝에 /wrap 한 번 호출
   ↓
1. 작업 요약 — 오늘 무엇을 했나 (1단락)
2. 멀티에이전트 분석 — 어떤 패턴이 반복됐나
3. 신호 누적 — signals/compound-log.json에 기록 (영구)
4. 임계치 알림 — 같은 패턴이 3번 쌓이면 "rules/에 박을 자리" 노티
5. 메모리 저장 — 다음 세션 시작 시 자동 참조 (CLAUDE.md memory)
```

본인이 의식 안 해도 **매 세션 끝에 패턴이 쌓이고**, 3회 도달하면 본인이 알 수 있게 신호가 온다. Notice → Fix가 자동으로 연결된다.

핵심 한 줄: **wrap이 Compound의 Notice 단계를 본인 대신 적어준다.**

---

**wrap이 없으면? 다른 자동화/구조화 방법**

wrap이 유일한 답은 아니다. 본인 환경에 맞는 다른 방법도 가능:

| 방법 | 어떻게 | 장점 | 단점 |
|------|------|------|------|
| **수동 회고 메모** | 매일 작업 후 SKILL.md `## 회고`에 1줄 기록 | 도구 필요 X, 본인 페이스 | 잊어버림. 일주일 후 안 적음 |
| **슬랙 리마인더** | 매주 금요일 오후 5시 슬랙봇이 "이번 주 반복 패턴 한 줄 적기" 알림 | 시간 차단·외부 자극 | 알림 무시 가능. 적는 건 사람 |
| **Notion DB** | "패턴" DB 만들고 매일 row 1개 추가 | 검색·필터·집계 자동 | 도구 따로 열어야 함. 마찰 |
| **캘린더 반복** | 매주 금요일 30분 회고 미팅 (본인 단독) | 시간 못 빠뜨림 | 미팅 자체가 부담 |
| **GitHub Actions** | git commit 시 패턴 분석 자동 실행 | 완전 자동 | 설정 진입 장벽 (개발자 영역) |
| **/wrap 스킬** | 세션 끝에 한 번 호출 | Claude Code 안에서 끝, 자동 누적 | wrap 스킬 설치·이해 필요 |

선택 기준: 본인이 매일 의식 안 하고 굴러가는 게 무엇인가. **마찰이 가장 작은 것이 답.** 비개발자 CSM 대부분에게는 `/wrap`이 가장 마찰 작다 (Claude Code 안에서 끝나서). 다만 슬랙 리마인더 + 수동 회고 메모 조합도 충분히 작동한다.

오늘 캠프 마지막 실습은 **wrap을 한 번 직접 호출**해서 어떤 출력이 나오는지 보고, 본인 워크플로우에 끼울지 결정하는 것.

---

**본인 Compound 파이프라인 — 1단·2단 만들기 실습**

오늘 끝낼 때 본인 손에 다음 두 가지가 있어야 한다.

**1단 — SKILL.md에 `## 회고` 섹션 추가 + /wrap 호출 약속**

본인이 어제 만든 SKILL.md 파일 하단에 `## 회고` 섹션을 추가한다.

```markdown
## 회고
- 2026-04-30: (오늘 첫 줄, 본인 wrap 시연 결과)
- 2026-05-07: ...
- 2026-05-14: ...
```

세션 종료 약속: **본인 스킬을 쓰는 모든 세션 끝에 `/wrap` 한 번 호출**. wrap 출력에서 가장 큰 신호 한 개를 `## 회고`에 1줄로 기록.

**2단 — 3회 반복 신호 → rules/ 자동 추가**

wrap이 같은 패턴을 3회 신호하면 임계치 알림이 온다. 그때 한 줄로 Claude에게 부탁:

```
"이 패턴(wrap이 알려준 것)을 본인 rules/required.md에 추가해줘.
이 스킬이 다음에 호출될 때 자동 반영되도록."
```

Claude가 commit + push까지 자동. 다음 사이클부터 본인이 의식 안 해도 자동 적용.

이게 Compound 파이프라인 1단·2단. **3주 운영하면 본인 SKILL.md가 본인이 명시적으로 쓴 적 없는 패턴까지 학습한 상태**가 된다.

**6축 자가진단**

4일 캠프의 6가지 축 전체를 O/X로 점검하는 체크리스트다.

| 축 | 점검 질문 | 0.5판 기준 |
|----|---------|----------|
| 1. 구조 | CLAUDE.md에 담당 고객·톤·금지어가 있는가? | 3섹션 이상 구체적으로 |
| 2. 맥락 | MCP 1개 이상 active 상태인가? | /mcp에서 active 확인 |
| 3. 계획 | Deep Interview 스펙 문서가 있는가? | 독자·입력·성공 기준 |
| 4. 실행 | 스킬 1개가 실제로 돌아가는가? | /skill 실행 성공 |
| 5. 검증 | 에이전트 피드백을 1번 이상 받았는가? | 지적·근거·제안 읽음 |
| 6. 개선 | 피드백을 rules/에 1건 이상 반영했는가? | commit 기록 확인 |

3개 이상 O면 4일 과정 0.5판 합격. 나머지는 2주 프로젝트에서.

**첫날 vs 마지막날 비교**

Day 1 B2에서 맥락 없이 돌린 결과를 기억하는가? 오늘 같은 요청을 다시 해보면:
- CLAUDE.md(축 1·2) + MCP(축 2) + 스킬(축 4)이 다 켜진 상태
- "이 고객 전용 결과"가 나온다

4일의 차이가 눈에 보인다.

**짝꿍 대화 가이드 (5분)**

짝꿍과 서로의 6축 체크리스트를 보면서:
- "어느 축이 O인가?" → 서로 확인
- "가장 약한 축은?" → 2주 프로젝트 방향
- "이 분야가 어렵다면 짝꿍에게서 뭘 배울 수 있나?" → 팀 시너지

### 3. CSM 일상 적용

**시나리오 1 — Notice 단계 실전**

리캡 메일 스킬을 3번 쓰고 나니 "다음 액션 날짜가 항상 빠진다"는 것을 발견했다. 이게 Notice다. rules/required.md에 "다음 액션 + 날짜 필수"를 추가하는 것이 Fix다. 4번째부터 날짜가 자동으로 들어간다.

**시나리오 2 — 6축 중 약한 곳 집중**

6축 체크에서 "검증(축 5)"이 X다. 에이전트 피드백을 한 번도 돌리지 않았다. 2주 프로젝트 첫 주에 까다로운 고객 에이전트를 내 스킬에 연결하는 것을 목표로 잡는다.

**시나리오 3 — Compound가 쌓이는 순간**

처음에는 "이 정도면 보낼 수 있는가?"를 5분간 판단했다. 3주 뒤에는 에이전트 확인 30초 + 내 판단 1분으로 줄어든다. 이것이 복리다.

---

### 4. 흔한 오해

**오해 1 — "스킬이 완성되면 Build→Use→Notice→Fix가 필요 없다"**

완성된 스킬은 없다. 고객이 바뀌고, 우리 팀 기준이 바뀌고, 새로운 금지어를 발견한다. 사이클은 끝나지 않는다. 다만 한 번의 반복에서 고치는 양이 점점 줄어든다.

**오해 2 — "6축이 모두 O면 완성이다"**

6축 O/X는 "지금 이 순간의 상태"를 보는 것이다. 완성의 의미가 아니다. "6개 O"인 상태에서도 더 나아질 수 있다. 기준은 "얼마나 나아졌는가"다.

**오해 3 — "Notice는 문제를 발견하는 것이라 부정적이다"**

Notice는 개선 재료를 발견하는 것이다. "이 부분이 계속 문제네"라는 발견이 스킬을 더 강하게 만드는 원료다. 문제를 발견했다는 것은 그 부분을 Fix할 수 있다는 뜻이다.
