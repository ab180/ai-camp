## EXPLAIN

### Generator vs Evaluator — 역할 분리 원리

```
AI에게 "네가 쓴 초안 네가 검토해줘"라고 하면?
→ 대부분 칭찬 위주. 자기 결과에 편향.

규칙: 만든 쪽과 점검하는 쪽이 같으면 검증이 아니라 자기 확신.
```

| 역할 | 특징 |
|------|------|
| **Generator (만드는 쪽)** | 내 스킬이 만든 초안. 나한테는 그럴듯해 보임. 맥락이 머릿속에 이미 있음 |
| **Evaluator (점검하는 쪽)** | 별도 에이전트. 고객 페르소나로 읽음. 채점 기준이 명시됨. 지적·근거·제안 포맷 |

### "까다로운 고객" 에이전트

페르소나 + 채점 기준 + 출력 포맷 = 에이전트

| 항목 | 내용 |
|------|------|
| **1. 페르소나** | 시니어 PM · 바쁜 구매 의사결정자 · 3년차 사용자. 기대 수준·바쁘기·소통 톤 명시 |
| **2. 채점 기준** | 결정사항 명시·다음 액션·책임자/기한·오해 소지·톤 적절성. 5~7개 항목 |
| **3. 출력 포맷** | 지적 → 근거 → 제안 3단 구조. 근거는 본문 인용. 제안은 한 문장 |

**피드백 예시**
```
[지적] 다음 액션이 모호합니다
[근거] "추후 공유드리겠습니다"라는 표현 — 언제인지 불명확
[제안] "이번 주 금요일까지 데이터팀 검토 후 회신드리겠습니다"로 수정
```

### 검증 실습 흐름 (8분)

```
내 결과물 1건을 까다로운 고객에게 넣기:

Claude Code에서:
  너는 아래 페르소나로 내 초안을 읽어줘.
  
  페르소나: 바쁜 시니어 PM. 구체적 다음 액션과 기한이 없으면 짜증냄.
  채점 기준: 결정사항 명시/다음 액션/책임자-기한/오해 소지/톤
  출력 포맷: 지적 → 근거 → 제안 (각 1-2문장)
  
  [내 초안 붙여넣기]
```

피드백을 읽고 메모해두세요. B2에서 3번 반복 규칙에 쓰입니다.
