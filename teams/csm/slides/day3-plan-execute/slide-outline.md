## B0 · 어제 만든 스킬 짝꿍 점검 (5장) · S05–S09

### S05 · 섹션 오프너 · Block 0 · ~15min
- 제목: 어제 만든 스킬 — 짝꿍이 한 번 봐주기
- 서브: 만든 사람은 자기 빈틈을 못 본다

### S06 · 왜 짝꿍 점검인가
- 2-col: 본인이 다시 봄 (같은 가정 안 → 빈틈 통과) vs 짝꿍이 봄 (다른 가정으로 충돌 → 빈틈 노출)
- foot: 이 구조가 B2 역할 분리(Generator vs Evaluator)의 사람 버전 — 다리 한 줄

### S07 · 짝꿍이 확인하는 3가지
- 3-row 카드:
  1. 입력값 가이드 명확? (이걸 줘 vs 막연)
  2. 출력 형태 정해짐? (불릿/단문/표)
  3. 빠진 항목? (당연히 있어야 하는데 없는 것)
- 메시지: 본인 도메인 모르는 짝꿍일수록 빨리 잡는다

### S08 · 짝꿍 호출 흐름 (4단계)
- 4-step:
  1. SKILL.md 교환 (Slack DM/GitHub)
  2. 짝꿍이 입력자로 한 번 호출
  3. 빈틈 메모 1-2줄 양방향
  4. 메모를 B1 인풋으로 가져감
- 시간: ~12분

### S09 · 메모 예시 + 다음 블록 연결
- 예시 메모 3개 (terra 카드):
  - "입력에 [참석자] 없어도 결과 나오는데 임의로 채워짐"
  - "출력이 매번 다른 형식 (불릿 → 단문)"
  - "다음 액션 담당자가 명시 안 됨"
- foot: 이 메모를 옆에 두고 B1으로 — 본인 케이스 #1에서 처음부터 의식

## B1 · 본인 케이스 사이클 반복 (8장) · S10–S17

### S10 · 섹션 오프너 · Block 1 · ~40min
- 제목: 사이클 반복 — 본인 케이스 1·2·(3)
- 서브: 70 → 90% 체감 + 본인 암묵지 식별

### S11 · 케이스 정하기 3기준
- 3-row:
  - (a) 반복 빈도 — 주 2회 이상
  - (b) 입출력 명확 — "~ 주면 ~ 나온다" 한 줄
  - (c) MCP 활용 가능 — Notion/Slack/캘린더 중 하나
- 흐름: 후보 3개 적기 → 짝꿍에게 "입출력 가장 명확한 게 뭐야?" → 답이 케이스 #1

### S12 · 1사이클 4단계 (복기 + 본인 손)
- 4-step (Day 2와 같은 흐름, 내용만 본인 것):
  1. 자연어 부탁 (B0 메모 옆에)
  2. SKILL.md 읽기 (실행 X)
  3. Clarify 4가지로 재질문
  4. 답 모아 재부탁 (90%)
- 메시지: 사이클은 동일, 내용만 본인 것

### S13 · 70%의 의미
- 핵심 카드: 70%는 실패가 아니라 의도된 출발점
- 3-cell: 1회 결과(70%) → Clarify(채움) → 재부탁(90%)
- 메시지: 한 번에 100%는 안 온다. 그 간격이 학습의 자리

### S14 · 케이스 #1 실습 ~15분
- 4-step 카드 (자연어 / SKILL.md / Clarify 4가지 / 재부탁+비교)
- 메모: 첫 부탁 결과 vs 재부탁 결과 — 무엇이 달라졌는가 1-2줄

### S15 · 케이스 #2 반복 + 패턴 ~15분
- 핵심: 케이스 #1에서 빠진 게 #2에서도 빠지나?
- 체크포인트: "나는 항상 [___]를 빠뜨린다" 1줄
- B0 짝꿍 메모와 비교 — 겹치면 패턴 확정

### S16 · 케이스 #3 (선택) + 패턴 추출
- 시간 되면 한 번 더
- 변화 신호: 처음 부탁부터 80-85%, 빠진 항목 줄어듦
- 패턴 1줄 정리 (SKILL.md `## 회고` 섹션 후보)

### S17 · 짝꿍 공유 5분
- 3 questions:
  1. 본인 항상 빠뜨리는 항목 vs 짝꿍 항목
  2. 겹치는 항목 / 다른 항목
  3. 짝꿍 SKILL.md에서 본인이 가져갈 1개

## B2 · Subagent 본격 (9장) · S18–S26

### S18 · 섹션 오프너 · Block 2 · ~35min
- 제목: Subagent 본격 — 3패턴 + 역할 분리 실습
- 서브: B0의 사람 짝꿍 = B2의 AI Evaluator (다리 한 줄)

### S19 · Subagent 개념
- 한 줄 정의: 메인 AI가 독립 AI를 일회성 호출 — 결과만 받는다
- 3특성: 독립 컨텍스트 / 병렬 처리 / 결과만 반환

### S20 · 3패턴 개요
- 패턴 1 병렬 검색 → 시간 단축
- 패턴 2 역할 분리 → 품질 향상 (오늘 실습)
- 패턴 3 컨텍스트 보호 → 메인 컨텍스트 절약

### S21 · 패턴 1 병렬 검색 상세
- before/after: 순차 150초 vs 병렬 30초
- CSM 예: 미팅 5건 동시 액션 추출

### S22 · 패턴 2 역할 분리 상세 (오늘 실습)
- Generator(작성) → Evaluator(검토) 다이어그램
- 핵심: 만든 사람은 자기 맹점을 못 본다 — B0과 같은 원리
- B0 다리: 사람 짝꿍 = Evaluator 역할

### S23 · 패턴 3 컨텍스트 보호 상세
- before/after: 100페이지 직접 vs Subagent + 핵심 3줄
- 3패턴 요약표 (목적·CSM 케이스)

### S24 · 역할 분리 실습 ~15분
- 3단계:
  1. Generator: B1 사이클 결과 1건 준비 (5분)
  2. Evaluator: 까다로운 고객 페르소나로 검토 요청 (5분)
  3. 검토 결과 메모 1-2줄 (3분)
- 관찰: Generator vs Evaluator 사이의 gap

### S25 · CSM 사용처 3케이스
- 미팅 리캡 마감 (병렬)
- 갱신 리스크 메일 (역할 분리)
- 분기 리뷰 자료 (컨텍스트 보호)

### S26 · 주의점 3가지 + 짝꿍 활용 대화
- 주의점: 맥락 공유 X / 비용 증폭 / 결과 합치기 책임
- 짝꿍 5분: 순차→병렬 후보 / 정보 모으는 작업 / 기다리며 딴 일 패턴

## B3 · Git/PR 협업 한 사이클 (5장) · S27–S31

### S27 · 섹션 오프너 · Block 3 · ~20min
- 제목: Git/PR 협업 — 어제 PR 받기 → 머지 → pull
- 서브: 어제 "올리기"에 더해 한 바퀴 완주

### S28 · 어제 vs 오늘 차이표
- 6-row 비교표:
  - clone (어제 ✓)
  - 파일 수정 → 코멘트 받기/답하기 (오늘)
  - commit (둘 다)
  - push → 같은 PR 자동 업데이트 (오늘)
  - PR 생성 → 짝꿍 승인 → squash merge (오늘)
  - (없음) → pull 동기화 (오늘 신규)
- foot: 5단계 다 굴러봐야 "이제 알겠다"

### S29 · 협업 사이클 다이어그램
- 5-step flow:
  1. 짝꿍 코멘트 ("출력 형태 가이드 모호")
  2. 답 + 수정 commit
  3. 같은 PR에 push (자동 업데이트)
  4. 짝꿍 승인 → squash merge
  5. main에서 git pull → log 확인
- 시간: 각 단계 1-5분

### S30 · 충돌(conflict) 5분 가이드 (옵션)
- 4-step (난이도 표시):
  1. GitHub 웹 "Resolve conflicts" 버튼
  2. `<<<<<<< ===== >>>>>>>` 마커 보기
  3. 어느 쪽 살릴지 결정 후 마커 제거
  4. "Mark as resolved" → "Commit merge"
- 메시지: 충돌은 정상 — 망가진 게 아님

### S31 · 새 PR 1개 (선택) + 머지 후 git log
- 명령어 카드:
  ```
  git checkout -b day3-cycle-pattern
  vi members/<이름>.md   # Day 3 패턴 1줄 추가
  git add . && git commit -m "feat: Day 3 사이클 패턴"
  git push -u origin day3-cycle-pattern
  # GitHub 웹에서 PR 생성, 짝꿍 리뷰어 지정
  ```
- foot: 머지 후 본인 git log 3개 commit — 어제 → 오늘 변화량 누적 시각화
